Development Strategy and Product Iteration 
Plan 
This section defines the product development lifecycle for CleanML — from the initial Minimum 
Viable Product through to a fully-featured AI-powered preprocessing platform. It provides a 
structured roadmap that explains not only what will be built, but in what order, and why that order is 
the correct engineering decision. 
The strategy is grounded in the principle that production software is not born complete. It is grown 
through deliberate, feedback-driven iteration. Each phase produces a working, deployable, and 
testable system. Each deployment generates real user signal. Each round of user signal drives the 
next engineering decision. 
 
🎯  Strategic Objective 
Ship a working product to real users as fast as possible. Learn what matters. Build that. Repeat. This 
is how every successful ML tool — from Weights & Biases to Hugging Face to Streamlit — was 
actually built.

MVP Development 
Core dataset cleaning — the minimum product that solves a real problem 
 
13.1.1 Purpose of the MVP 
The Minimum Viable Product is the smallest version of CleanML that delivers genuine, standalone 
value to a real user. It is not a prototype or a demo. It is a deployable, functional system that solves 
a concrete, painful problem: turning a raw, messy dataset into a clean, ML-ready dataset without 
writing a single line of code. 
The MVP deliberately excludes NLP preprocessing, text vectorisation, AutoML integration, and all 
other advanced capabilities. Those features are valuable — but they are not the core problem. A 
data science student who cannot impute missing values or remove duplicates does not benefit from 
TF-IDF vectorisation. The MVP solves the foundational problem first. 
 
ℹ️  MVP Design Principle 
Every feature included in the MVP must pass this test: Does removing this feature make the product 
useless for the core user? If yes, it stays. If no, it moves to Phase 2 or later.

13.1.3 MVP Architecture Constraints

The MVP architecture is intentionally simplified relative to the full system design. These constraints 
reduce initial complexity without creating technical debt, because the architecture is designed to be 
extended in place. 
• Synchronous processing only — Celery async workers are not required until dataset sizes 
exceed typical MVP usage 
• Redis is optional — session state is managed via signed JWT tokens stored in browser 
cookies in MVP 
• PostgreSQL is not required — no job history, no user accounts, no audit log in Phase 1 
• Local filesystem storage — no S3 integration required; Parquet files stored on the server 
instance 
• Single worker process — uvicorn with 2 workers is sufficient for MVP load levels

DEPLOY Early Deployment 
Ship the MVP publicly before it is perfect — then learn 
 
13.2.1 Why Deploy Early 
One of the most common mistakes in software product development is waiting until the product is 
"finished" before deploying it. There is no finished. There is only shipped and unshipped. An 
unshipped product generates zero user signal, zero feedback, and zero validation that the design 
assumptions are correct. 
CleanML is deployed publicly as soon as the MVP feature set passes integration tests and 
produces correct output on at least three representative datasets. Imperfect UX, missing features, 
and rough edges are expected and acceptable at this stage. The purpose of early deployment is not 
to impress users — it is to learn from them. 
 
ℹ️  Deployment Trigger Criteria 
The MVP is ready to deploy when: (1) file upload works for CSV, Excel, and JSON; (2) profiling 
produces correct statistics on a dataset with mixed types and nulls; (3) at least two cleaning 
operations work correctly; (4) one-hot encoding and StandardScaler produce correct output; and (5) 
the exported CSV matches the cleaned DataFrame. 
 
13.2.2 MVP Deployment Stack 
The initial deployment uses managed hosting platforms that require zero infrastructure 
management. This allows the engineering team to focus on product quality rather than server 
operations during the validation phase.

13.2.4 What Early Deployment Validates 
Every assumption made during the design and development phases is exactly that — an 
assumption. Early deployment converts assumptions into evidence. The following product 
hypotheses are tested by real users during the early deployment phase: 
• Upload reliability: Do real-world datasets (not just clean test fixtures) upload and parse 
without errors? 
• Profile accuracy: Do users trust the statistics shown? Are there edge cases (mixed-type 
columns, unusual encodings) that break the profiler? 
• UX clarity: Can a data science beginner complete a cleaning workflow without reading 
documentation? 
• Performance: Are wait times acceptable for typical dataset sizes? Where do users abandon 
the workflow? 
• Feature priority: What do users ask for that is missing? What do they ignore that was built?

ITERATE Iteration and System Stabilisation 
Fix what is broken and sharpen what is blunt — before building more 
 
13.4.1 The Purpose of an Iteration Sprint 
After the first feedback cycle, the team enters a dedicated iteration sprint whose sole purpose is to 
fix, stabilise, and sharpen the MVP — not to add new features. This constraint is deliberate. Adding 
features before the core workflow is solid creates a system with a wide surface area and a weak 
foundation. 
The iteration sprint transforms the MVP from a system that works in ideal conditions into a system 
that works reliably under real-world conditions. This includes unusual datasets, unexpected user 
behaviour, edge cases in encoding logic, and performance under realistic load. 
 
13.4.2 Typical Iteration Improvements 
Based on patterns observed in similar ML tools during early deployment, the following categories of 
improvements are consistently required after first user contact: 
 
UI Clarity Improvements 
• Replace generic error messages ("Something went wrong") with specific, actionable ones 
("Column age contains non-numeric values and cannot be scaled. Convert it to numeric first.") 
• Add inline documentation tooltips explaining what each preprocessing operation does and 
when to use it 
• Clarify the step progression — users should always know what step they are on, what step is 
next, and how to go back 
• Improve the column selector component — early testing consistently reveals confusion 
around multi-column selection UX 
• Add empty state messages — when a dataset has no missing values, the cleaning panel 
should say so rather than showing an empty form 
 
Performance Improvements 
• Profile endpoint optimisation — if profiling a 100k-row dataset takes more than 4 seconds, 
introduce sampling (compute stats on a representative 10k-row sample and note this in the 
UI) 
• Parquet preview reads — replace full DataFrame loads for preview endpoints with pyarrow 
slice() calls to avoid loading entire files 
• Frontend bundle size — audit Next.js bundle; lazy-load heavy components (chart libraries, 
large tables) that are not needed on initial paint 
• Connection pooling — if the backend shows latency spikes, review pandas operations for 
unnecessary DataFrame copies and add uvicorn worker scaling 
 
Error Handling Improvements

13.4.3 Stabilisation Criteria 
The iteration phase is complete — and Phase 2 development may begin — when the following 
criteria are met: 
1. No P0 bugs open for more than 48 hours over a 2-week period 
2. Core workflow completion rate (upload → export) is above 70% in session analytics 
3. Median profiling time for a 50k-row dataset is under 3 seconds 
4. Upload success rate is above 95% for valid CSV, Excel, and JSON files 
5. User feedback sentiment (collected in feedback widget) is net positive on core usability

13.5.1 What Changes in Phase 2 
Phase 1 delivers a tool. Phase 2 begins to deliver a platform. The distinction matters. A tool does 
what you tell it to do. A platform does what you need it to do — sometimes before you know you 
need it. Phase 2 introduces intelligence, automation, and richer feedback to the preprocessing 
workflow. 
These features are introduced only after Phase 1 is stable because they depend on the foundation 
being correct. An automatic preprocessing suggestion is worthless if the underlying imputation logic 
it recommends is broken. Dataset quality scoring is misleading if the statistics it uses are 
inaccurate. Phase 2 is built on top of Phase 1, not instead of it. 
 
💡  Phase 2 Design Principle 
Every Phase 2 feature must make the core Phase 1 workflow faster, smarter, or more 
understandable. Features that add complexity without improving the core workflow are deferred to 
Phase 3.

Feature 2: Dataset Quality Score 
A composite 0–100 quality score is computed from four weighted dimensions and displayed 
prominently on the profiling dashboard. The score gives users an immediate quantitative sense of 
how much cleaning work is needed — and tracks improvement as cleaning operations are applied. 
 
# quality_scorer.py 
def compute_quality_score(df: pd.DataFrame) -> dict: 
    n_rows, n_cols = df.shape 
 
    # Completeness: what % of cells have values (not null) 
    completeness = 1 - (df.isnull().sum().sum() / (n_rows * n_cols)) 
 
    # Uniqueness: what % of rows are not duplicates 
    uniqueness = len(df.drop_duplicates()) / n_rows 
 
    # Consistency: what % of columns have a single consistent dtype 
    # (mixed-type columns penalise this score) 
    consistent_cols = sum(1 for col in df.columns 
        if df[col].apply(type).nunique() == 1) 
    consistency = consistent_cols / n_cols 
 
    # Validity: what % of numeric values are within 3 std devs 
    numeric_cols = df.select_dtypes(include="number") 
    if len(numeric_cols.columns) > 0: 
        z_scores = np.abs((numeric_cols - numeric_cols.mean()) / 
numeric_cols.std()) 
        validity = (z_scores < 3).values.mean() 
    else: 
        validity = 1.0 
 
    # Weighted composite score 
    score = ( 
        completeness * 0.40 +    # Missing values have highest impact 
        uniqueness   * 0.25 +    # Duplicates degrade model training 
        consistency  * 0.20 +    # Type inconsistency causes pipeline failures 
        validity     * 0.15      # Outliers reduce generalisation 
    ) * 100 
 
    return { 
        "overall": round(score, 1), 
        "completeness": round(completeness * 100, 1), 
        "uniqueness": round(uniqueness * 100, 1), 
        "consistency": round(consistency * 100, 1), 
        "validity": round(validity * 100, 1), 
    }

Feature 3: One-Click Pipeline Generation 
Based on the automatic preprocessing suggestions, the system offers a "Generate Pipeline" button 
that applies all recommended operations in the correct sequence with a single click. The generated 
pipeline is fully editable — users can remove or modify individual steps before applying. This 
reduces the time from upload to first cleaned dataset from minutes to seconds for typical use cases. 
 
Feature 4: Pipeline Visualisation UI 
The PipelineVisualiser component (introduced in Phase 1 as a step log) is enhanced in Phase 2 to 
display a proper graphical pipeline diagram — a horizontal flow of labelled, coloured nodes 
connected by directional arrows. Each node shows the operation type, affected columns, and a 
summary of the change (e.g., "—1,240 rows" for deduplication, "+8 columns" for one-hot encoding). 
The diagram is exportable as SVG and included in the pipeline export package.

TEST 2 Second Feedback Cycle 
Validate that Phase 2 features are useful, usable, and correct 
 
13.6.1 Why a Second Feedback Cycle 
Every time new features are added to a product, new failure modes are introduced. Automatic 
preprocessing suggestions may be wrong for unusual datasets. The quality score formula may not 
reflect what users intuitively consider "quality." The pipeline visualiser may be confusing if the graph 
becomes too complex. A second feedback cycle validates that Phase 2 features add value rather 
than confusion. 
 
13.6.2 Specific Evaluation Questions for Phase 2 
Phase 2 Feature Evaluation Question 
Preprocessing Suggestions Do users trust the suggestions? Do they apply them? When do they 
override them and why? 
Quality Score Does the 0–100 score match users' intuition about their dataset' quality? Is 
the per-dimension breakdown actionable? 
One-Click Pipeline Does applying all suggestions in one click produce acceptable results, or 
do users need to customise every time? 
Pipeline Visualiser Can users read and understand the pipeline diagram without explanation? 
Do they use it for debugging? 
 
13.6.3 Feedback-to-Roadmap Mapping 
Feedback from the second cycle is used to make a binary decision for each Phase 2 feature: ship 
as-is, iterate, or remove. No feature survives on the roadmap based on engineering preference 
alone — user evidence is required. 
• High usage, positive feedback → ship as-is, consider extending in Phase 3 
• High usage, negative feedback → iterate the implementation; the need is real but the 
execution is wrong 
• Low usage, positive feedback → document better; users who find it love it but cannot find it 
• Low usage, negative feedback → candidate for removal; may be solving the wrong problem

13.7.1 When Phase 3 Begins 
Phase 3 begins when Phase 1 and Phase 2 are both stable, well-tested, and generating positive 
user signal. There is no fixed calendar date for this transition. The trigger is product maturity, not 
elapsed time. A system that is six months old but still generating P1 bugs from the MVP feature set 
is not ready for Phase 3 development. 
The features in Phase 3 are deliberately advanced — they require additional Python dependencies 
(spaCy, gensim, PyCaret), additional infrastructure (heavier worker containers, more Redis 
memory), and more complex frontend components. Adding them before the foundation is stable is 
the surest way to create an unmaintainable codebase.

13.7.3 Phase 3 Infrastructure Changes 
Phase 3 features require infrastructure upgrades that were intentionally deferred from Phase 1. 
These upgrades are now justified by real usage data and revenue/scale requirements. 
• Celery async workers: Required for NLP preprocessing (spaCy is CPU-intensive) and 
AutoML training jobs which can run for minutes 
• S3 / GCS cloud storage: Required for collaborative features and pipeline versioning — files 
must persist across container restarts and be accessible by multiple workers 
• PostgreSQL: Required for user accounts, template library, job history, and pipeline version 
storage 
• Heavier worker containers: spaCy + gensim + PyCaret require 2–4 GB RAM per worker; 
worker containers are upsized independently of the API containers 
• CDN for static assets: As the user base grows, static Next.js assets and chart images 
should be served from a CDN to reduce origin server load

PHILOSOPHY Product Development Philosophy 
The principles that govern how CleanML is built and evolved 
 
13.8.1 The Iterative Development Model 
CleanML is built according to an iterative product development model. This model is not a 
methodology — it is a discipline. It requires the team to continuously resist the urge to add features 
before the existing features are solid, and to continuously resist the urge to delay deployment until 
the product is "ready." Products are never ready. They are only better or worse than they were 
before. 
 
Idea  →  MVP  →  Deploy  →  Feedback  →  Iterate  →  Phase 2  →  Feedback  →  
Phase 3 
 
13.8.2 Why This Approach Reduces Risk 
Every software project carries two categories of risk: technical risk (can we build this?) and product 
risk (should we build this?). Most development teams focus on technical risk and ignore product 
risk. The iterative model addresses both. 
 
Risk Type How the Iterative Model Mitigates It 
Wrong features built Phase 1 scope is validated by real users before Phase 2 begins. Features 
that users do not use are not extended. 
Architecture not fit for 
purpose 
Early deployment under real load surfaces architectural weaknesses 
before they are deeply embedded in a large codebase. 
Performance problems in 
production 
Real datasets are structurally different from test fixtures. Early deployment 
exposes performance problems on data the team did not anticipate. 
Scope creep and schedule 
overrun 
Explicit phase boundaries with defined completion criteria prevent 
unbounded feature expansion within a phase. 
Wasted engineering effort Phase 3 features are not designed in detail until Phase 1 and Phase 2 are 
stable. No time is spent architecting AutoML integration while the basic 
CSV upload is still broken. 
User adoption failure Early deployment generates early adopters. Early adopters become 
advocates. A product that ships late ships without an existing community. 
 
13.8.3 The Role of User Feedback in Engineering Decisions 
In a traditional waterfall development process, engineering decisions are made by architects and 
product managers based on assumptions about user needs. In the iterative model, engineering 
decisions are made by the same people — but based on evidence from real users rather than

This does not mean that users dictate the technical architecture. Users describe symptoms; 
engineers diagnose causes and design solutions. A user who says "the app is slow" is not 
requesting a Redis cache or a Parquet optimisation — they are describing an experience. It is the 
engineer's job to translate that experience into a technical decision. The iterative model ensures 
that those translations are made in response to real signals rather than hypothetical ones. 
 
🔑  Core Principle 
Build for the user you have, not the user you imagined. The user you imagined lives in your design 
document. The user you have lives in your session analytics, your feedback widget, and your 
support inbox. One of them is real. 
 
13.8.4 Iteration Cadence 
The following cadence governs all development work at CleanML from MVP through Phase 3: 
 
Cadence Activity 
Every 2 weeks Iteration sprint — review feedback queue, prioritise P0/P1 items, ship fixes and 
small improvements 
Every 4–6 weeks Feature sprint — deliver one Phase 2 or Phase 3 feature; must pass integration 
tests before deployment 
Every 8 weeks Feedback review — structured review of session analytics, error rates, and user 
interview insights; adjust roadmap if warranted 
On every deployment Smoke test — automated end-to-end test suite (Playwright) that verifies the 
core upload → clean → export workflow in the production environment 
Continuously Error monitoring — Sentry alerts are reviewed within 24 hours; P0 bugs are 
patched and deployed within 48 hours 
 
13.8.5 Summary 
The development strategy for CleanML can be expressed in a single sentence: build the smallest 
useful thing, deploy it, listen to the people who use it, and use what you hear to decide what to build 
next. 
This is not a simple strategy. It requires discipline to defer features that seem important, honesty to 
acknowledge when a design assumption was wrong, and humility to accept that the user's 
experience of the product is more important than the engineer's intention for it. 
The phases described in this section are not a fixed plan. They are a structured starting point. Real 
development will deviate from this plan in ways that cannot be predicted — and that deviation is not 
a failure. It is evidence that the iterative model is working.
